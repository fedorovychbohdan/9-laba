# Test Strategy — v1.0
> **Див. також:** Стратегія тестування розроблена для верифікації нефункціональних вимог, вказаних у [SSD](../architecture/SSD.md).

## 1. Загальний підхід
Забезпечення якості (QA) для REST API сервісу бронювання квитків базується на стратегії **багаторівневої перевірки**. Це дозволяє виявляти помилки на різних етапах життєвого циклу розробки — від окремих функцій до високого навантаження.

## 2. Рівні та види тестування

Для системи впроваджені наступні типи перевірок:

| Тип тестування | Об'єкт перевірки | Інструментарій |
| :--- | :--- | :--- |
| **Unit Testing** | Окремі функції бізнес-логіки (розрахунок ціни, валідація дат). | Go Test / PyTest |
| **Integration Testing** | Взаємодія між мікросервісами (Booking ↔ Payment) та робота з PostgreSQL. | Docker Compose, Testcontainers |
| **API Testing** | Коректність HTTP-статусів та структур JSON-відповідей для API v1. | Postman / RestAssured |
| **Load Testing** | Симуляція пікового навантаження (1000+ запитів/хв). | k6 / JMeter |
| **Security Testing** | JWT-автентифікація, захист від SQL-ін’єкцій та вразливостей OWASP. | OWASP ZAP / Snyk |

## 3. Критерії прийняття (Acceptance Criteria)
* Усі Unit-тести мають статус "Passed".
* Покриття коду (Code Coverage) основними тестами становить не менше 80%.
* Середній час відповіді під навантаженням не перевищує **250 мс** (згідно з [SSD](../architecture/SSD.md)).

## 4. Автоматизація
Всі тести інтегровані в CI/CD пайплайн. Будь-яка зміна в коді автоматично запускає набір регресійних тестів перед злиттям (merge) у головну гілку.
