# Контрактна API-документація (Ticketing System)

Цей документ описує роль, призначення та структуру API системи бронювання квитків згідно з підходом API-first.

## 1. Роль та призначення API
* **Тип API:** Internal (внутрішнє).
* **Цільова аудиторія:** Frontend-застосунок (SPA), мобільні клієнти.
* **Призначення:** API надає контракт для взаємодії frontend-частини з сервером для керування пошуком рейсів, бронюванням місць та переглядом статусів замовлень.

## 2. Формування API (API-first Definition)
Нижче наведено детальний опис ендпоінтів, методів та структур даних, визначених до початку реалізації.

### 2.1. Пошук доступних квитків
* **Метод:** `GET`.
* **Ендпоінт:** `/api/tickets`.
* **Призначення:** Отримання списку рейсів за заданими фільтрами.
* **Структура відповіді (200 OK):**
    ```json
    {
      "count": 1,
      "items": [
        {
          "id": "T-102",
          "route": "Kyiv - Prague",
          "price": 1500,
          "available_seats": 24
        }
      ]
    }
    ```
* **Коди помилок:** `400 Bad Request` (невірні параметри пошуку).

### 2.2. Створення бронювання
* **Метод:** `POST`.
* **Ендпоінт:** `/api/bookings`.
* **Структура запиту (Request Body):**
    ```json
    {
      "ticket_id": "T-102",
      "passenger_name": "Ksenia",
      "seat_number": 12
    }
    ```
* **Структура відповіді (201 Created):**
    ```json
    {
      "booking_id": "B-5501",
      "status": "confirmed"
    }
    ```
* **Коди помилок:** * `401 Unauthorized` (необхідна авторизація).
    * `409 Conflict` (місце вже зайняте).

### 2.3. Оновлення даних бронювання
* **Метод:** `PATCH`.
* **Ендпоінт:** `/api/bookings/{id}`.
* **Призначення:** Зміна параметрів існуючого бронювання (наприклад, зміна місця).
* **Коди помилок:** `404 Not Found` (бронювання не знайдено).

### 2.4. Скасування бронювання
* **Метод:** `DELETE`.
* **Ендпоінт:** `/api/bookings/{id}`.
* **Відповідь:** `204 No Content` (успішно видалено).
* **Коди помилок:** `403 Forbidden` (скасування неможливе за умовами тарифу).

## 4. Swagger UI та автоматизація

### 4.1. Візуалізація та тестування
На основі створеної специфікації було згенеровано інтерфейс **Swagger UI**. Він дозволяє переглядати документацію в інтерактивному форматі та виконувати тестові запити до API без використання сторонніх клієнтів.

### 4.2. Генерація документації з коду
У межах DaC-підходу передбачено можливість генерації `openapi.yaml` безпосередньо з коду застосунку (наприклад, через анотації в Spring Boot або Go), що забезпечує актуальність документації на всіх етапах розробки.
