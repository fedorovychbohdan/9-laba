    System Specification Document (SSD) — v1.0

    Навігація: Повернутися на головну | Переглянути план розробки (Roadmap)

    1. Призначення системи

    Сервіс призначений для автоматизації повного циклу продажу квитків: від пошуку доступних заходів до фіскалізації оплати. Система виступає центральною ланкою між організаторами подій та кінцевими споживачами, забезпечуючи надійне керування інвентарем місць у реальному часі та запобігаючи дублюванню замовлень.

    2. Основні функції (Functional Requirements)

    Для зручності відстеження кожна функція має свій унікальний ідентифікатор:

    ID Функція Опис
    FR-01 Пошук та фільтрація Користувач може отримувати список подій за категоріями, датами та локаціями через REST-запити.
    FR-02 Керування інвентарем Відображення актуального статусу кожного місця (вільно/заброньовано/продано) з мінімальною затримкою.
    FR-03 Механізм бронювання Тимчасове блокування місця для конкретного сеансу користувача на 15 хвилин до моменту завершення оплати.
    FR-04 Обробка платежів Інтеграція з зовнішніми шлюзами (Stripe/LiqPay) для підтвердження транзакцій.
    FR-05 Генерація квитків Формування унікального електронного квитка з QR-кодом після успішної оплати.

    3. Обмеження та припущення

    Обмеження

    • Система не підтримує офлайн-продаж квитків (тільки через API).
    • Технічне обмеження: Один користувач не може забронювати більше 6 квитків за одну транзакцію для запобігання перепродажу.

    Припущення

    • Зовнішні платіжні шлюзи мають власну доступність не нижче 99.9%.

    4. Нефункціональні вимоги (NFR)

    Категорія Вимога Показник (KPI)
    Продуктивність Швидкість обробки Не менше 1000 запитів на хвилину; середній час відповіді (latency) до 250 мс.
    Доступність Час роботи Не менше 99.9% (SLA); не більше 8.7 годин простою на рік.
    Безпека Захист даних Використання TLS 1.3 та JWT-токенів для автентифікації кожного API-запиту.
    Масштабованість Навантаження Горизонтальне масштабування (Auto-scaling) при збільшенні навантаження в 5 разів.
