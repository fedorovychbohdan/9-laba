# Infrastructure Specification Document (ISD) — v1.0
> **Див. також:** Конфігурація серверів та мережі базується на структурі мікросервісів, описаній у [Технічному дизайні (SDD)](SDD.md).

## 1. Середовище розгортання
Система розгортається у хмарному середовищі (AWS / Google Cloud) з використанням оркестратора контейнерів **Kubernetes (K8s)**. Це забезпечує повну ізоляцію мікросервісів (`Booking`, `Payment`, `Event`) у власних Docker-контейнерах та дозволяє ефективно керувати обчислювальними ресурсами.

## 2. Основні інфраструктурні компоненти

Для забезпечення стабільної та відмовостійкої роботи API використовуються наступні компоненти:

| Компонент | Технологія | Функціональне призначення |
| :--- | :--- | :--- |
| **Load Balancer** | Nginx / AWS ALB | Розподіл вхідного трафіку між репліками сервісів для запобігання перевантаженню. |
| **Database Cluster** | PostgreSQL | Основне реляційне сховище з реплікацією **Master-Slave** для захисту від збоїв. |
| **In-memory Store** | Redis | Кешування подій та механізм **TTL-блокування** для тимчасового бронювання. |
| **Message Broker** | RabbitMQ | Забезпечення асинхронних черг (наприклад, для `Notification Service`). |
| **Object Storage** | S3 (AWS/Minio) | Зберігання статичного контенту: афіші подій та згенеровані PDF-квитки. |

## 3. Принципи масштабування

Система спроектована для обробки "вибухового" навантаження (старту продажів популярних подій) за наступними принципами:

* **Horizontal Pod Autoscaling (HPA):** Kubernetes автоматично створює нові екземпляри (Pods), якщо завантаження CPU або RAM перевищує **70%**.
* **Database Scaling:** Використання **Read Replicas** для обробки великої кількості GET-запитів (`/events`), що знімає навантаження з Master-вузла.
* **Content Delivery Network (CDN):** Кешування зображень афіш на серверах, розташованих ближче до кінцевого користувача, для прискорення завантаження інтерфейсу.
