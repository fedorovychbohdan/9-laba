# Test Plan — v1.0
> **Див. також:** Цей план деталізує сценарії тестування для вимог із [SSD](../architecture/SSD.md) згідно із загальною [Стратегією тестування](test-strategy.md).

## 1. Об'єкт тестування
Об'єктом тестування є функціональні модулі системи бронювання квитків, що відповідають за конкурентний доступ до ресурсів та стабільність під навантаженням.

## 2. Детальний план перевірки функцій

| ID Тесту | Функція для перевірки | Опис перевірки та очікуваний результат |
| :--- | :--- | :--- |
| **TP-01** | Конкурентне бронювання | **Перевірка обробки 1000 запитів на одне місце одночасно.** Очікуваний результат: систему забронює місце лише для першого успішного запиту, інші отримають помилку "Seat already taken". |
| **TP-02** | Валідація TTL (Time-to-Live) | Створення броні та очікування 15 хвилин без оплати. Очікуваний результат: статус місця автоматично змінюється з "Locked" на "Available". |
| **TP-03** | Цілісність транзакції | Емуляція збою під час підтвердження оплати. Очікуваний результат: система виконує Rollback і не генерує квиток без успішного статусу транзакції. |

## 3. Ресурсне забезпечення
- **Середовище:** Staging-кластер Kubernetes.
- **Інструменти:** Скрипти на базі `k6` для генерації паралельних запитів (TP-01).
- **Дані:** Тестова база подій з наповненням понад 10 000 місць.

## 4. Графік виконання
Тестування проводиться автоматично в рамках кожного релізу (CI Pipeline). Сценарій **TP-01** (навантаження) запускається окремо перед кожним великим оновленням архітектури.
