    Single Source of Truth (SSoT) та правила оновлення

    Навігація: Повернутися на головну | Цей документ визначає ієрархію джерел істини для всіх технічних специфікацій проекту.

    1. Розподіл джерел істини

    Щоб інформація не дублювалася і завжди була актуальною, визначено головні документи для кожної сфери:

    • Вимоги (функціональні та нефункціональні): Джерелом істини є SSD.md. Усі вимоги мають унікальні ID (наприклад, FR-01).
    • Архітектура та компоненти: Джерелом істини є SDD.md. Тут описується внутрішня будова мікросервісів та їх взаємодія.
    • Інфраструктура: Джерелом істини є ISD.md. Опис серверів та Kubernetes.
    • Тести та якість: Джерелом істини є Traceability Matrix. Самі тести не вигадують нові вимоги, а лише посилаються на ID з SSD.

    2. Правила оновлення документації

    Документація вважається частиною коду, тому для неї діють такі правила:

    • Зміна вимог: Якщо замовник змінює функцію, спочатку оновлюється SSD.md, а потім — тести.
    • Зміна архітектури: При зміні API-ендпоінтів обов'язково оновлюється SDD.md у тому ж самому Pull Request (коміті), що і код.
    • Заборона копіювання: Заборонено копіювати опис вимоги з SSD в інші файли. Замість цього використовується лише посилання на ID (наприклад, "див. FR-01") .

    3. Модель версійності та актуалізація

    Для підтримки актуальності документів використовується модель семантичного версіонування, синхронізована з релізами системи:

    • Major зміна (наприклад, v1.0 -> v2.0): Повний перегляд архітектури, зміна технологічного стеку або фундаментальна зміна бізнес-моделі сервісу.
    • Minor зміна (наприклад, v1.0 -> v1.1): Додавання нових функціональних вимог (FR), розширення API-ендпоінтів або оновлення інфраструктурних схем.
    • Patch зміна (наприклад, v1.0.1): Виправлення помилок у тексті, уточнення технічних описів або оновлення внутрішніх посилань між документами.

    Принцип актуалізації: Документація версії v1.0 відповідає стабільному релізу системи. При будь-якій зміні API або вимог, нова версія документації створюється та публікується одночасно з випуском оновленого програмного коду.
