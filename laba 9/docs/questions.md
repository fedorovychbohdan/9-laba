# Контрольні питання (ЛР№8)

## 1. У чому полягає принцип Documentation as Code (DaC)?

Принцип **Documentation as Code** полягає в застосуванні підходів розробки ПЗ до створення документації. Це включає:

* Використання текстових форматів (Markdown, AsciiDoc).
* Зберігання документів у системах контролю версій (Git) разом із кодом.
* Автоматизовану збірку документації у вигляді веб-сайтів (MkDocs).
* Процес рецензування змін (Code Review) для забезпечення якості контенту.

## 2. Що таке Single Source of Truth (SSoT)?

**Single Source of Truth (Єдине джерело істини)** — це архітектурний принцип, за якого кожен елемент даних зберігається в одному-єдиному місці. 

* Усі інші документи посилаються на це першоджерело.
* Це виключає дублювання інформації та ризик виникнення суперечностей.

## 3. Чому Markdown є базовим форматом для документації?

Markdown є стандартом де-факто через такі переваги:

* **Людиночитаність:** Текст зрозумілий навіть без рендерингу.
* **Версійність:** Git чітко відстежує зміни по рядках.
* **Швидкість:** Легко редагується в будь-якій IDE (наприклад, PyCharm).
* **Конвертованість:** Легко перетворюється на HTML або PDF.

## 4. Як забезпечується актуальність документації у DaC-підході?

Актуальність підтримується завдяки інтеграції в життєвий цикл розробки:

* **Синхронність:** Зміни в документацію вносяться разом із кодом в одному Pull Request.
* **Автоматизація (CI/CD):** При оновленні гілки `main` сайт автоматично перезбирається.
* **Прозорість:** Документація знаходиться поруч із кодом, що полегшує її підтримку.

# Контрольні питання (ЛР№9)

### 1. Чому API-документація є контрактом?
API-документація виступає **контрактом**, оскільки вона фіксує чітку домовленість між надавачем послуги (Backend) та її споживачем (Frontend/Mobile). Вона визначає:

* Точні адреси ресурсів (ендпоінти) та методи доступу до них.
* Обов'язкові структури даних для вхідних запитів та відповідей.
* Очікувану поведінку системи у випадку помилок (HTTP статус-коди).

Це гарантує, що обидві сторони розробки працюють синхронно, не порушуючи цілісність системи при оновленнях.

### 2. У чому перевага API-first?
Підхід **API-first** передбачає проектування контрактів до початку написання коду[cite: 7, 32]. Головні переваги:

* **Паралельна розробка:** Команда фронтенду може почати роботу, використовуючи "заглушки" (mocks) на основі специфікації, не чекаючи готовності бекенду.
* **Зменшення вартості помилок:** Недоліки архітектури виявляються на етапі обговорення специфікації, а не під час інтеграції готового коду.
* **Покращений UX для розробників:** API стає більш логічним та структурованим, оскільки проектується з думкою про кінцевого споживача.

### 3. Яку роль відіграє Swagger UI?
**Swagger UI** є інструментом візуалізації та інтерактивної взаємодії зі специфікацією OpenAPI:

* **Документування:** Перетворює технічний YAML/JSON файл у зручний веб-інтерфейс.
* **Пісочниця (Sandbox):** Дозволяє виконувати реальні тестові запити до API безпосередньо з браузера за допомогою кнопки "Try it out".
* **Зменшення порогу входження:** Допомагає новим розробникам швидко зрозуміти можливості системи без вивчення вихідного коду.

### 4. Навіщо потрібен Storybook?
**Storybook** використовується для візуального документування та ізольованої розробки UI-компонентів:

* **Опис станів:** Дозволяє зафіксувати вигляд компонента у різних сценаріях (базовий стан, помилка, завантаження, неактивний стан).
* **Ізоляція:** Компоненти розробляються окремо від складної бізнес-логіки застосунку.
* **Спільна мова:** Служить живим гайдом для дизайнерів та розробників, забезпечуючи візуальну цілісність проекту.

### 5. Чому документація має бути частиною Definition of Done?
Включення документації до **Definition of Done (DoD)** є критичним для підтримки її актуальності:

* **Актуальність:** Функція не вважається виконаною (Done), доки її опис не з'явиться в документації.
* **Синхронізація:** Це запобігає ситуаціям, коли код працює одним чином, а документація описує інший.
* **Автоматизація:** Це стимулює впровадження перевірок (валідація специфікацій, збірка сайту) безпосередньо в CI/CD пайплайни.
