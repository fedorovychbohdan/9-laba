# Software Design Document (SDD) — v1.0
> **Див. також:** Дана архітектура розроблена на основі вимог, визначених у [Специфікації системи (SSD)](SSD.md).

## 1. Загальна архітектура
Для системи обрано **мікросервісну архітектуру**. Це дозволяє:
* **Незалежно масштабувати компоненти**: наприклад, виділяти більше ресурсів для `Event Service` під час пікових навантажень.
* **Забезпечити високу відмовостійкість**: збій у сервісі повідомлень не зупиняє процес продажу квитків.

## 2. Основні компоненти системи

Система декомпонована на наступні функціональні модулі:

| Компонент | Роль та відповідальність | Технологічний стек (приклад) |
| :--- | :--- | :--- |
| **API Gateway** | Маршрутизація, JWT-автентифікація, Rate Limiting. | Nginx / KrakenD |
| **Event Service** | Менеджмент каталогу подій та схем залів. | Go, PostgreSQL, Redis |
| **Booking Service** | Логіка бронювання, перевірка доступності, TTL Lock. | Go, PostgreSQL |
| **Payment Service** | Інтеграція зі Stripe/LiqPay, обробка транзакцій. | Python / Go |
| **Notification Service** | Асинхронна відправка Email/Push повідомлень. | Node.js / Go |

## 3. Взаємодія між компонентами

### Синхронна взаємодія (REST/gRPC)
Використовується для критичних операцій, що потребують негайної відповіді:
* `Client` -> `API Gateway` (REST JSON)
* `API Gateway` -> `Booking Service` (gRPC для швидкості)

### Асинхронна взаємодія (Message Broker)
Використовується для подій, що обробляються у фоновому режимі через **RabbitMQ** або **Kafka**:
1. **Payment Service** публікує подію `Order.Paid`.
2. **Booking Service** підписується на подію для фіналізації статусу місця.
3. **Notification Service** підписується на подію для відправки квитка користувачу.

# Створення тимчасового бронювання (TTL 15 хв)
